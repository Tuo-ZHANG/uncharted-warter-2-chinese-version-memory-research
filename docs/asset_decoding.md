# 《大航海时代2》图像资产逆向解析与底层机制笔记

这是一份关于 90 年代 DOS 游戏（以光荣《大航海时代2》为例）在图像编码、文件归档、内存管理以及底层寻址方面的硬核技术复盘。

---

## 一、 图像位深（Bit Depth）与 1920 字节的数学逻辑

**位深（Bit depth）**决定了计算机为了记录**一个像素点的颜色**所分配的比特（Bit）数量。它直接决定了画面的色彩细腻程度。

### 1. 核心计算公式
可显示的颜色数量 = 2 ^ 位深

### 2. 头像数据的精准控容
在《大航海时代2》中，海员头像采用了标准的 **3位深（3-bit）** 机制：
* 颜色数：2^3 = 8 色。
* 像素尺寸：80 * 64 像素。
* **体积计算**：
  80 * 64 * 3 bits = 15360 bits
  15360 bits / 8 = 1920 字节

这种“位平面（Bit Planar）”的设计配置意味着每 3 个字节控制连续的 8 个像素，将 3 个字节相同比特位上的 0 和 1 组合成 0~7 的颜色索引。这种非标准的私有魔改格式，导致现代通用图像软件（如 Photoshop、7YUV）在没有专用逆向渲染算法的情况下无法直接读取。

## 二、 光荣私有归档格式：LS11 的演进

在二进制文件中，前几个字节的**幻数（Magic Number）**用于标识文件身份。`LS11` 是光荣在 90 年代初自研的私有压缩归档引擎。

### 1. 版本演进历史
* **`LS10` (Version 1.0)**：早期初版（如《三国志3》），算法相对单纯，压缩率较低。
* **`LS11` (Version 1.1)**：《大航海时代2》的主力格式。引入了**“包内自带 256 字节自定义字典”**的微操，专门针对特定文件的色彩进行优化，大幅提升压缩率。
* **`Ls12` (Version 1.2)**：DOS 末期（如《三国志5》）的微调版，优化了位流读取和索引表结构。

## 三、 内存受限下的“即用即解”流转机制

在 90 年代 DOS 系统（著名的 640KB 常规内存限制）以及 1.44MB 软盘环境下，游戏引擎绝对不会采用“整体读入”或“写入本地临时文件”的策略。

### 游戏运行时的流式处理步骤：
1. **触发事件**：玩家触发 NPC 交互。
2. **精准定位**：查阅常驻内存的微型目录，获取目标头像的绝对偏移量（Offset）与压缩大小。
3. **流式读入**：通过硬件级的文件寻道（File Seeking），磁头直接跳到指定位置，仅读取几百字节的压缩数据到内存缓冲区 A。
4. **内存解压**：在内存中将其解压为 1920 字节的裸数据到缓冲区 B，随后立即释放缓冲区 A。
5. **直写显存**：将缓冲区 B 按照位平面算法转换成色彩索引，直接写入显存（如 `0xA000:0000`），随后立即释放缓冲区 B。

## 四、 逆向深挖：为什么不用“字典”检索？

在现代开发中，使用键值对字典（如 Python 的 `dict`）来存储动态偏移量是常识。但在 1993 年的硬件环境下，光荣程序员采用了更高级的 **O(1) 数组直接寻址**。

### 1. 放弃现代字典（Hash Map）的原因
* **内存开销严重**：哈希表需要应对冲突，维护链表指针（Pointer Chasing）会浪费几 KB 的宝贵内存空间。
* **CPU 计算瓶颈**：执行哈希函数（乘除法与位运算）在当时的 286/386 处理器上会消耗宝贵的时钟周期。
* **内存碎片化**：离散的动态内存分配容易导致 DOS 系统崩溃。

### 2. 光荣的底层微操：固定步长数组
由于 `LS11` 的目录中，每条记录（Record）被严格限定为 **12 字节**（4字节压缩大小 + 4字节解压大小 + 4字节绝对偏移量）。

整个目录在内存中是一个连续的、固定长度的数组。当游戏需要查询第 N 号资产时，CPU 根本不需要遍历，也不需要算哈希，而是直接进行指针偏移：
目标记录内存地址 = 目录起始地址 + (N * 12)

在汇编层面，乘以 12 只需要极速的位移指令即可完成。这种将数据结构固化在二进制格式中的微操，实现了纯硬件级别的寻址速度。

## 五、 衍生发现：864 字节文件的数学解密

在逆向解包过程中，除了 1920 字节的头像文件外，还会解出大量的 **864 字节** 的 `.bin` 文件。利用同样的位平面数学逻辑可以完美倒推其尺寸：

* **总比特数**：864 字节 * 8 = 6912 bits
* **总像素数**（3位深）：6912 bits / 3 = 2304 像素
* **尺寸开平方**：根号(2304) = 48

**结论**：这批资产是游戏中 **48 * 48 像素、8色** 的正方形小图标，对应游戏中的**道具物品（Items）**以及**地理发现物（Discoveries）**的配图。通过将解码器中的尺寸校验从硬编码（80x64）改为动态识别（48x48），即可完美渲染这部分资产。
